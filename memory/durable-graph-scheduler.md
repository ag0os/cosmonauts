# Durable Runtime Phase 3: Graph Scheduler

## What Was Built
The riskiest plan of the durable orchestration track (Plan 3 of 4): the **smallest useful local durable graph scheduler**, sitting on the Plan-1/Plan-2 substrate. Two new modules — `lib/durable-runtime/scheduler.ts` (in-process drain loop, lease/heartbeat/backend/concurrency/signal orchestration) and `lib/durable-runtime/scheduler-state.ts` (pure reconciliation, zero IO) — plus store-owned graph/scheduler-state/heartbeat/diagnostic primitives on `FileRunStore`. It loads a persisted graph, marks dependency-satisfied steps `ready`, leases and heartbeats running work, detects stale leases **from persisted heartbeat age**, retries-or-blocks by explicit policy, finalizes the run from terminal step outcomes, and supports **bounded, opt-in** parallelism behind an explicit limit. Sequential-first (default `maxParallelSteps: 1`), in-process (D-010), file-backed. 20 behaviors (B-001..B-020); integration report `correct`, zero findings. **No frontend migration** — Drive and chains stay on their legacy loops until Plan 4; the B-020 contract test pins that the Drive adapter is *not* a Plan-3 scheduler backend.

## Key Decisions
- **In-process library, not a daemon (D-010).** `runDurableGraphScheduler(options)` drains until the run is terminal, blocked, cancelled, or has only fresh externally-owned running work. Pure reconciliation lives in `scheduler-state.ts`; IO/backend loop in `scheduler.ts`. Tests prove durable recovery with no process management.
- **Leases are fields, never a status.** `StepRecord.lease` / `SchedulerState.leasesByStepId` carry ownership while `StepStatus` stays `running`. The canonical `RunStatus`/`StepStatus` unions were preserved exactly — B-001 rejects any `queued`/`waiting`/`leased` drift. All Plan-3 contracts (`StepLease`, `StepHeartbeat`, `RetryPolicy`, `SchedulerState`, `RunGraph`, scheduler result/exit-reason types) extend canonical types *additively in place*, no renames.
- **`graph.json` is immutable/topology only; `steps/<id>/step.json` is the sole mutable authority.** `RunGraph` carries `RunGraphStep[]` (id/runId/title/kind/backend/dependsOn/inputArtifacts) — it deliberately *excludes* `status`, `result`, `latestAttemptId`, `lease`, `heartbeat`, `retryPolicy`. This kills the dual-source recovery bug class (PR-004). Originally `RunGraph.steps: StepRecord[]`; the reviewer forced the split.
- **No default hard timeout (D-004).** Liveness is heartbeat-based, not wall-clock. `hardTimeoutMs`/`idleTimeoutMs`/`staleHeartbeatMs` are all optional policy. Long-idle work with fresh heartbeats stays `running` indefinitely (B-005); staleness is only inferable when `staleHeartbeatMs` is explicitly configured.
- **Scheduler persists only through `RunStore`.** Added `readRunGraph`/`writeRunGraph`/`read|writeSchedulerState`/`read|writeStepHeartbeat`/`listStepRecords`/`appendDiagnostic`. The scheduler has *zero* `node:fs` imports (B-002 inspects sources). Keeps the SQLite/remote-store migration path open and respects the store/scheduler boundary.
- **Sequential-first, parallelism last (D-007).** Bounded parallelism (B-012/B-016) was implemented only after sequential scheduling, leases, heartbeats, stale/fresh recovery, canCommit safety, retry, cancellation, and crash recovery were all green. Implementation order mattered: correctness is the riskiest part, parallel mutable work stays constrained until worktree policy is explicit.

## Patterns Established
- **Stale detection reads PERSISTED heartbeat age, never empty memory.** A restarted scheduler boots with empty in-memory lease/heartbeat maps — those are *not* evidence that work stopped. Staleness compares persisted `StepHeartbeat.at` against `staleHeartbeatMs`. Empty in-memory state must never fabricate a default lease/heartbeat and rerun work (B-006/B-010). This is the load-bearing anti-pattern the whole recovery algorithm exists to forbid.
- **Crash recovery splits on backend capabilities — block-committed vs leave-running.** When a restart finds a nonterminal `running` step with no terminal attempt evidence:
  - **canCommit + no safe resume → block** (`blockPotentiallyCommittedRunningSteps`, B-014). A `canCommit: true`/`canResume: false` backend may have committed externally before `BackendHandle.result` settled and before `writeStepAttemptRecord`. Retrying would duplicate committed work. Block with diagnostic `potentially_committed_step_blocked` unless `retryPotentiallyCommittedSteps: true` is explicitly set (an unsafe opt-in that must be surfaced in diagnostics).
  - **Fresh heartbeat + non-resumable → leave running, don't duplicate** (B-015). Return `exitReason: "waiting_for_fresh_external_work"` with `externally_owned_fresh_running_work`. Zero `prepare`/`start`, no new attempt, no fabricated lease. When `staleHeartbeatMs` is absent, staleness can't be decided — leave it running.
  - **canCommit: false (idempotent) → stale retry is safe** after stale detection, because there's no committed external work to duplicate.
- **Recovery promotes terminal attempts before considering any rerun (B-011).** Crash after `attempt-001` got `endedAt`+`result` but before `step.json` flipped from `running`: read `latestAttemptId`, promote the step to its terminal state, then never call `backend.start`. The crash-recovery algorithm runs this *before* selecting ready work.
- **Retry = append a NEW attempt, never erase.** `nextAction: "retry"` requeues by explicit transition and writes `attempt-002`; `attempt-001` stays readable with its evidence; `latestAttemptId` advances. Effective attempts: `step.retryPolicy.maxAttempts` → else `RunPolicy.retryLimit + 1` → else `1` (no retry by default). Status moving back to `ready`/`running` is legal *only because* a fresh attempt is appended. Completed steps are terminal — `completed -> running` never happens.
- **Step records are the sole mutable authority on conflict (B-017/B-018).** If `graph.json` carries legacy mutable fields, ignore them and emit `ignored_graph_mutable_state`. A graph step with missing/corrupt `step.json` does *not* default to runnable — it blocks before execution (`missing_step_record`/`corrupt_step_record`). A graph claiming terminal/running state can never cause a duplicate start or suppress a `step.json` terminal/blocking state.
- **Shared-worktree mutable concurrency cap (B-016).** Concurrent starts under `worktree.mode === "shared"` are allowed *only* for backends that are explicitly safe: `isolatedFromHostSource: true && canCommit: false`. Committing/non-isolated backends cap effective concurrency to `1` with `shared_worktree_mutable_concurrency_capped`, even when `maxParallelSteps` is higher. Isolated non-committing backends are the positive parallel case.
- **D-006 unknown-vs-success.** A `RunGraphSchedulerBackend` returns `StepResult`; any malformed/non-`StepResult` output normalizes to `outcome: "unknown"` and **blocks** (never trusted as success). Unknown results and exhausted retries block instead of advancing dependents (B-008). Unknown/missing backend name also blocks rather than skipping.
- **Signal cancellation preserves evidence (B-019).** `RunGraphSchedulerOptions.signal` is invocation-local (not a user-facing mutating controller — D-009). For `canCancel: true`: call `cancel` once, write a terminal `cancelled` attempt, release lease, finalize. For backends that can't confirm cancellation: do *not* fabricate terminal state or release the lease — preserve open attempt/heartbeat as externally-owned evidence and emit `cancellation_not_supported`.
- **Finalizer failure is a retryable `finalizer` step, not a task failure.** The generic enums add no `finalization_failed` member; a finalizer step fails with `nextAction: "retry"` plus enough evidence to resume. Plan 4's Drive compatibility layer maps that retryable failed finalizer onto Drive's `finalization_failed` outcome.

## Gotchas & Lessons
- **The reviewer round caught the load-bearing bug *in the plan*, pre-implementation.** PR-001 (committed-work crash window) exposed that B-009/B-011 only proved no-duplication *after* terminal evidence was already persisted — the actually-dangerous window is crash *after* an external commit but *before* `writeStepAttemptRecord`. Resolution added B-014 + `retryPotentiallyCommittedSteps`, and the fake commit-side-effect counter must fire *before* `BackendHandle.result` settles. Five of six PR findings (committed window, fresh-heartbeat recovery, shared-worktree cap, graph/step dual-source, backend-contract erasure) became new behaviors or contract changes. Run the adversarial plan review before tasking a high-risk scheduler.
- **Current Drive adapters dictated the safety design.** `claude-cli` and `cosmonauts-subagent` are `canCommit: true`; all three are `canResume: false`; default policy has no `staleHeartbeatMs`. The recovery rules (block committed, leave fresh-external running) are shaped precisely by those real capabilities — not hypothetical ones.
- **`OrchestrationBackend<unknown, unknown>` is an unsafe erasure.** Typing the registry as `RunGraphSchedulerBackend = OrchestrationBackend<SchedulerStepInput, StepResult>` makes the Drive adapter (`<BackendInvocation, BackendRunResult>`) non-assignable — B-020 pins this with a `@ts-expect-error` guard *and* greps scheduler sources for `lib/driver/backends/*` imports. The Drive `BackendInvocation` builder (needs `runId`/`promptPath`/`workdir`/`projectRoot`/`taskId`/`parentSessionId`/`planSlug`/`eventSink`) is deferred to Plan 4; the generic scheduler must never invent those fields.
- **`readSchedulerState` normalizes the Plan-1 `{}` scaffold at the store boundary only** (empty `readyStepIds`, empty maps). Recovery must still read persisted step/attempt/heartbeat records — an empty normalized scheduler state is a reconstruction aid, never proof that no lease/heartbeat exists.
- **Recompute the ready queue from `step.json` statuses + dependencies**; use `SchedulerState.readyStepIds` only as a deterministic-ordering snapshot, not the source of truth. Terminal step records override stale scheduler snapshots *and* graph-embedded mutable fields.
- **Keep `backoffMs` inert if it adds complexity.** The plan explicitly allows storing `RetryPolicy.backoffMs` in state and deferring timed wakeups rather than growing a daemon scheduler — broad delayed scheduling was a non-goal.
