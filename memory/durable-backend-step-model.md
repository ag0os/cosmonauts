# Durable Runtime Phase 2: Backend and Step Model

## What Was Built
Plan 2 of the durable orchestration track: it makes the runtime model *concrete* around Drive task execution while Drive still owns its loop — no scheduler, no graph compiler, no chain migration, no CLI behavior change. Three deliverables: (1) a generic `OrchestrationBackend` adapter contract in `lib/durable-runtime/backends.ts` plus a Drive `Backend`→adapter wrapper that preserves invocation behavior exactly; (2) generic `StepRecord`/`StepAttemptRecord` persistence projected from existing Drive events (`lib/driver/durable-steps.ts`) — task steps with backend identity, dependencies, attempts under `steps/<stepId>/attempts/<attemptId>/`, and terminal `StepResult`; (3) Drive finalization phases (source commit, task status, state commit) modeled as `finalizer` step records without weakening `finalization_failed` recovery. The load-bearing safety property is D-006: malformed/missing backend reports persist `outcome: "unknown"`, never silent success, in attempt + step records *and* the actual normalized `step_completed.result`. Behaviors B-001..B-011; 2286 tests green, artifact conformance 11/0, fallow audit clean.

## Key Decisions
- **The Drive adapter is intentionally NOT assignable to the scheduler backend.** The scheduler is hard-typed `OrchestrationBackend<SchedulerStepInput, StepResult>` (aliased `RunGraphSchedulerBackend`); the Drive adapter is `OrchestrationBackend<BackendInvocation, BackendRunResult>`. Plan 2 deliberately produces backend-specific output — `BackendHandle.result` returns Drive's `BackendRunResult`, and Drive's terminal `StepResult` is synthesized by `durable-steps.ts` from parsed reports/finalization events, *not* inside the wrapper — so the wrapper never duplicates Drive report parsing. The B-020 bridge (a real scheduler-facing adapter that builds `SchedulerStepInput` and parses to `StepResult`) is deferred to Plan 4 and pinned by a `@ts-expect-error` test.
- **`OrchestrationBackend.name` was narrowed from the architecture's `string` to `KnownBackendName`** (`codex` | `claude-cli` | `cosmonauts-subagent` | `shell-command`). The Drive adapter narrows further to a 3-member `Extract<...>` (`DriveOrchestrationBackendName`) — `shell-command` has no Drive task adapter, it is the finalizer/verifier identity only.
- **`DriverRunSpec.backendName` is authoritative for persisted backend identity** — used for `StepRecord.backend.name`, adapter `options.name`, and `RunRecord.metadata.configuredBackendName`. `spawn_started.backend` / `ctx.backend.name` stays *legacy telemetry* (and normalized `step_started.backend` compat). `Backend.name` is intentionally `string` and tests use fakes like `fake-backend`; a mismatch records a `drive_backend_identity_mismatch` diagnostic but never widens `KnownBackendName` or writes the fake name into a step record (PR-008).
- **Backend-name + spec types live ONLY in `lib/durable-runtime/types.ts`; `backends.ts` imports them and owns only execution contracts** (`BackendCapabilities`, `BackendContext`, `PreparedStep`, `BackendHandle`, `OrchestrationBackend`). Putting `BackendSpec` next to `OrchestrationBackend` would force a cycle, since `types.ts` already owns `RunPolicy.defaultBackend` and `StepRecord` (PR-009).
- **`unknown` is kept as a compatibility `BackendName` for unconfigured generic run-policy defaults only** — `BackendName = KnownBackendName | "unknown"`. Plan-1 `FileRunStore` already creates runs with `{ name: "unknown" }`; forcing a real backend would lie in persisted records (PR-005). `unknown` has no adapter and must never appear on a Drive task/finalizer step.
- **Step records are evidence only — do NOT retarget Plan-1 normalized finalization events.** Finalizer `StepRecord`s carry the new recovery model; `pending-finalization.json` / `run.completion.json` / CLI resume stay authoritative. Retargeting `task_finalization_failed` → finalizer step IDs would change `run_watch` output, so it's deferred to frontend migration (PR-004).

## Patterns Established
- **Pinned per-backend capability flags** (`DRIVE_BACKEND_ORCHESTRATION_CAPABILITIES`): all three are `canResume:false`, `canCancel:false`, `emitsMachineReport:true` (machine report is the Drive prompt/report *contract*, not a backend-native guarantee). `codex` = `canCommit:false`, `isolatedFromHostSource:true`; `claude-cli` = `canCommit:true`, isolated; `cosmonauts-subagent` = `canCommit:true`, **NOT** isolated (runs in-host). `shell-command` = `canCommit:true`, not isolated, `emitsMachineReport:false`.
- **Unsupported ops are explicit, not faked.** `resume`/`cancel` throw `UnsupportedDriveBackendOperationError` rather than pretending to work — current Drive cancellation flows through the pre-invocation `AbortSignal`, not a runtime control.
- **D-006 enforced via an in-memory latest-result map *before* file writes.** The projector keeps `latestResults` by taskId so a file-write failure never loses result context; `event-stream.ts` passes that context into `normalizeDriverEvent` for `task_done` so the normalized `step_completed.result` honors `unknown`. `durable-events.ts` falls back to Plan-1 success only for pure-unit/legacy callers with no context.
- **`status: "completed"` + `outcome: "unknown"` is a legitimate, pinned combination.** When a malformed-report task later emits legacy `task_done`, step *status* becomes `completed` (Drive lifecycle compat) but `result.outcome` stays `unknown` and `nextAction` is `wait_for_human`/absent — **never** `continue`. Future scheduler logic must inspect `StepResult`, not status alone.
- **Deterministic finalizer step IDs:** `finalizer-source-commit-<taskId>`, `finalizer-task-status-<taskId>`, `finalizer-state-commit`. Finalizer backend identity is `{ name: "shell-command", options: { drivePhase } }`. Passed/skipped → `outcome: "success"` (`nextAction: continue`); failure → `status: "failed"`, `nextAction: "retry"`, with a `pending-finalization` artifact ref. A finalization failure does **not** flip the behavioral task step to failed.
- **Attempts-under-steps layout, append-never-overwrite.** `steps/<stepId>/attempts/<attemptId>/{attempt.json, output.md, result.json}`; `step.json` holds current status + a `latestAttemptId` pointer. Attempt IDs are deterministic (`attempt-001`, `attempt-002`, …) derived by listing existing attempts, so retry/resume appends. `output.md` carries parsed-report evidence (or `ParsedReport.raw` for unknown reports), not full raw stdout. Unsafe `stepId`/`attemptId` are rejected before any write escapes the run dir.
- **Two distinct verification vocabularies are mapped separately (PR-007).** `ParsedReport.verification.status` `pass/fail/not_run` → runtime `pass/fail/skipped`; legacy `verify` DriverEvent `started/passed/failed` is handled separately (`started` = activity only). Report file changes `created/modified/deleted` → runtime `added/modified/deleted`; do **not** fabricate `renamed`.
- **Durable-only failure hooks for resume paths that emit no terminal event.** Source-commit, task-status, and state-commit resume failures write `run.completion.json` but no terminal `DriverEvent` today; Plan 2 adds durable-only recorder calls (not new legacy event JSON) so resume still appends a retryable finalizer attempt.

## Gotchas & Lessons
- **The `@ts-expect-error` test is the load-bearing guardrail for the deferred bridge.** It proves a `Map<string, RunGraphSchedulerBackend>` cannot hold a Drive adapter, AND greps `scheduler.ts`/`driver/backends/types.ts` to confirm the scheduler never imports `driver/backends` or `BackendInvocation`. If someone "fixes" the type error by widening the scheduler input, the architectural seam silently collapses — keep the error expected.
- **`StepRecord.status: completed` is NOT proof of success and was flagged as a misread risk.** Pin `nextAction` non-`continue` on every unknown result and document that schedulers must read `StepResult`. This is the single most subtle invariant in the plan.
- **Resumed runs rewrite the active spec to the remaining task slice** — deriving dependencies from `DriverRunSpec.taskIds` at resume time would erase deps on already-completed earlier tasks. Persist the *original* order in `RunRecord.metadata.driveTaskIds` at initial creation and never overwrite it on resume; fall back to the active slice (with a diagnostic) only for pre-Plan-2 runs lacking the metadata.
- **Durable persistence failures must stay diagnostics, never behavior failures.** Project steps *after* the legacy event write/publish succeeds; a failed `steps/.../step.json` write emits `drive_durable_step_write_failed` and is never converted into `EventLogWriteError`, task-blocked, run-abort, or finalization-failed. Keep the in-memory result map alive even when file persistence fails.
- **The unhandled state-commit resume seam was the review's top finding (PR-006).** Direct `commitFinalState` failures emit a `finalize` failed event, but a `skipped/no_changes` result followed by `acceptExternalStateCommit` rejection (pending task not `Done`, dirty task files, or unchanged `HEAD`) goes straight to `writeStateFinalizationFailure` with no terminal event. Every "this path writes completion JSON but emits no event" branch needs an explicit durable-only recorder call, or B-009's "resume always records a retryable attempt" promise is a lie.
- **QM round 1 caught two real regressions a 2286-test happy-path suite missed:** the fallow audit flagged unused type exports + duplicated finalizer-attempt logic + complexity, and the UX reviewer caught ordinary successful task completions gaining step-result fields in normalized events — the only legitimate normalized change is B-006's malformed→`unknown` correction. Keep the legacy success-completion event shape byte-identical except for that one correction.
- **Boundary direction is gated, not just asserted:** `lib/durable-runtime/*` must import nothing from Drive/CLI/domains/tasks/prompts/backend-impls; the Drive↔runtime bridge lives only in `lib/driver/backends/orchestration-adapter.ts` and `lib/driver/durable-steps.ts`. `lib/driver/types.ts` derives its `BackendName` via `Extract` from the runtime union (one source of vocabulary) using `import type` only.
- **Do not stub future backend names.** `package-cli`, `nested-run`, and `approval` are documented future names only — the dead-code gate rejects speculative adapters/exports for them. Only `shell-command` was added in wave 1.
