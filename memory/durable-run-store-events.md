# Durable Runtime Phase 1: Run Store and Normalized Events

## What Was Built
The foundation of the durable orchestration track: a generic, file-backed run/step/event store (`lib/durable-runtime`) plus a normalized `OrchestrationEvent` model, extracted from Drive's bespoke event log **without changing any Drive behavior**. Drive now dual-writes: its legacy `events.jsonl` stays the authoritative stream for `watch_events`/status/list/resume, while a Driveâ†’normalized adapter also appends canonical envelopes to a sidecar `orchestration-events.jsonl`. Two read-only observation helpers (`run_status`, `run_watch`) summarize/page the normalized stream by sequence cursor. No scheduler, no backend adapter, no execution change â€” this plan only lays the storage substrate and proves it round-trips a real Drive run. Behaviors B-001..B-017; ~2272 tests green, artifact conformance 17/17.

## Key Decisions
- **`orchestration-events.jsonl` sidecar, not root `events.jsonl`.** The target architecture reserves run-root `events.jsonl` for the normalized stream, but legacy Drive already owns that file (`watch_events` hard-codes it; Drive specs default `eventLogPath` to it). Migrating it now would change `watch_events` structured output and resume/status assumptions, violating the zero-behavior-change constraint (D-003). The sidecar is an **authorized wave-1 compatibility exception recorded in the architecture record's Storage Layout** â€” not a local plan deviation. `RunRecord.eventsPath` points at the sidecar.
- **Terminal events stay exactly canonical; rich evidence moves to preceding activity.** `step_blocked`/`step_failed`/`run_failed` carry only `reason`; only `step_completed`/`run_completed` carry `StepResult`/`RunResult`. All Drive-specific detail (progress, contradicted paths, retryable flags, finalization phase/task/commit, source outcome) lives in a preceding `step_tool_activity` event or a controller diagnostic. Extending terminal variants locally would fork the source-of-truth union before scheduler work begins (blocker finding PR-001).
- **The Drive adapter lives in `lib/driver/`, not `lib/durable-runtime/`.** `durable-events.ts` is the *only* module importing both `lib/driver/types.ts` and `lib/durable-runtime/*`. Dependency direction is inward: generic runtime never knows about Drive. Boundary-conformance gate enforces no `lib/durable-runtime/*` import of `lib/driver/*`, `cli/*`, `domains/*`, prompts, or task modules.
- **Read-only observation only.** Ship `run_status`/`run_watch`; do NOT register `run_pause`/`run_resume`/`run_cancel`/`run_intervene` (D-009). `watch_events` stays a legacy Drive reader, untouched â€” normalized observation is additive.
- **Plan-1 takes subsets of the canonical contracts.** `RunRecord` includes a minimal `RunPolicy` (reportInference, optional cost/token/timeout limits, `defaultBackend` as open data, `worktree` defaulting to `{mode:"shared"}`); `StepRecord` omits leases/heartbeats/retries/attempts. Use the canonical *names* and *extend* later (Plan 2 fills `defaultBackend` and populates Drive task/finalizer step records) â€” never rename.

## Patterns Established
- **`seq`/timestamp envelope as the storage contract.** Every event is wrapped `{ seq, timestamp, runId, event }` where `seq` is a store-assigned monotonic per-run integer and `timestamp` is ISO-8601. `run_watch`/`run_status`/recovery read by `seq`, never by JSONL line position. The returned cursor is the latest valid seq â€” **except** when a `limit` truncates the page, where it advances only to the last *returned* event (a paging bug fixed in QM remediation; otherwise unread events get skipped). `seq` survives store reopen (continues from persisted max, not a line count).
- **Multi-event mapping with evidence-before-terminal ordering.** `normalizeDriverEvent` may return several normalized events for one legacy event; when a legacy event carries both rich evidence and a terminal transition, evidence events are emitted *first* in the array so store-assigned `seq` preserves the "activity â†’ terminal" relationship.
- **Construction-safe, lazy, failure-isolated dual-write sink.** Constructing the durable sink writes nothing and creates no `run.json`. Run-record creation/adoption happens lazily *inside* the sink, after the legacy sink has accepted the event. Dual-write order is compatibility-first: await the legacy sink exactly as today, then attempt durable setup + normalized append, then swallow/report any normalized failure as a diagnostic â€” a normalized error must never become an `EventLogWriteError` or block the Drive run/task outcome.
- **Failed preflight must close the step.** Drive's preflight-failed path returns `{status:"blocked"}` with no legacy `task_blocked` event. The translator maps failed `preflight` to `step_tool_activity(details)` then canonical `step_blocked{reason}` (from stderr / branch-mismatch / fallback) so no `step_ready` is left without a terminal event â€” the event stream alone must reconstruct terminal state.
- **Single pure status summarizer.** `summarizeRunStatus(record, events)` in `controller.ts` is the only normalized status logic; the file store's `readStatus` delegates to it rather than duplicating. Terminal normalized events are source of truth and win over a stale `RunRecord.status`; when they disagree the summary exposes both via `statusSource`.
- **Path-safety on every store boundary, including `loadRun`.** Scope/run/step IDs and all run-owned paths (`graphPath`, `eventsPath`, `artifactsDir`, `schedulerStatePath`) are validated/normalized to resolve *inside* the scoped run dir; traversal IDs are rejected before any outside file is touched. QM hardened `loadRun` to re-validate paths from a (possibly corrupted) `run.json` so a tampered record can't escape the run directory.
- **Status classification stays a closed lookup** (`statusFromEvent`, `isTerminalStatus`, `isTerminalStepStatus` in `status.ts`) â€” terminal-vs-non-terminal is one switch/predicate, not scattered conditionals.

## Gotchas & Lessons
- **The store interface must not bake in the file layout.** A later SQLite/remote-coordinator store has to stay possible, so `RunStore` is scope-aware (`{scope, runId}`) and backend/scheduler/CLI-renderer-agnostic â€” the file paths are an implementation detail of `file-store.ts`, not part of the contract.
- **Characterize compatibility BEFORE extracting.** B-008/B-009/B-011 were written first and **pass against current behavior** when normalized sidecar files are manually seeded (status/list/watch already ignore non-legacy files); only genuinely new dual-write/helper plumbing starts red. Don't assume every characterization test should be red â€” re-phrasing this saved chasing phantom failures (PR-006).
- **Prove the real round-trip, not just fixture writes.** The load-bearing seam is "Drive loop creates the sink â†’ adapter adopts `run.json` â†’ `run_status`/`run_watch` read that exact `eventsPath` sidecar." B-016 runs the actual Drive loop with a fake backend, then follows `RunRecord.eventsPath` and asserts the controller reports correct terminal state â€” splitting proof across write-only and read-only tests left this integration untested (PR-004).
- **Eager run-record setup would break Drive before it starts.** Drive constructs event sinks *before* entering the run loop; if durable adoption ran eagerly and failed, Drive would die before the legacy sink emits its first event. Lazy + failure-isolated is mandatory, and B-017 negatively proves a setup failure before the first event leaves the Drive run intact (PR-005).
- **`unknown` must stay `unknown`.** Plan 1 does not reinterpret or harden Drive's report parsing â€” a malformed/missing report records `unknown`, never silently normalized to success (D-006). Report hardening is post-production.
- **Legacy-only events return no normalized event â€” don't fabricate.** `lock_warning`/`plan_completion_candidate` have no run-level advisory variant in the architecture; the translator returns a `legacyOnly` diagnostic rather than inventing a `stepId`, backend, artifact, or terminal event. Forcing every legacy event into the normalized stream pollutes the generic contract.
- **The clean-context review + QM pass earned its keep.** Beyond the seven plan-review blockers, the QMâ†’fixer chain surfaced and fixed a real `run_watch` cursor-truncation bug (skipped unread events) and a `loadRun` path-escape hole from a corrupted `run.json` â€” neither visible in the happy-path suite. Re-verify each finding with a failing-first test; the QM raises real issues but its own remediation under-fixes.
